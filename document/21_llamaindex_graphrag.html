Why LlamaIndex for GraphRAG?

프로덕션 레벨의 지식 그래프 기반 RAG 시스템을 구축하기 위해 LlamaIndex가 제공하는 5가지 핵심 아키텍처 이점

PropertyGraphIndex 패러다임

기존의 단순한 노드-엣지 연결을 넘어, 노드와 엣지 모두에 풍부한 메타데이터(Properties)를 저장할 수 있는 최신 객체입니다. 이를 통해 텍스트 원본, 엔티티 설명, 출처 문서 ID 등을 그래프에 직접 내장할 수 있습니다.

  • 노드에 텍스트와 벡터 임베딩 동시 저장
  • 관계(Edge)에 가중치나 출처 문서 정보 부여 가능

Vector + Graph 완벽 하이브리드

사용자의 질문을 벡터로 변환하여 의미론적 유사도가 높은 진입 노드(Entry Nodes)를 찾은 뒤, 해당 노드와 연결된 엣지를 따라 멀티홉(Multi-hop) 탐색을 수행하는 하이브리드 검색을 기본적으로 지원합니다.

Vector Search
의미 기반 진입점 탐색
Graph Traversal
관계망 확장 탐색
Rich Context
정확한 LLM 답변

Developer & Production Benefits

무한한 커스텀 및 모듈화

MS GraphRAG의 블랙박스 구조와 달리, 추출기(Extractor), 리트리버(Retriever), 프롬프트를 레고 블록처럼 교체할 수 있습니다. 자체 정의한 온톨로지(Ontology)를 주입하여 도메인 특화 추출이 가능합니다.

폭넓은 Graph DB 생태계

앞서 고민하신 Kuzu, Memgraph, NebulaGraph를 비롯해 Neo4j까지 GraphStore 인터페이스 하나로 손쉽게 스위칭이 가능합니다. 인프라 종속성이 없습니다.

점진적 업데이트 (Upsert)

새로운 문서가 추가되었을 때 전체 파이프라인을 재실행할 필요 없이, 기존 그래프에 새로운 노드와 관계만 효율적으로 덧붙일 수 있습니다. 프로덕션 환경의 비용과 시간을 기하급수적으로 절감합니다.

llamaindex_graphrag_example.py
Python 
# 1. 원하는 Graph DB 인스턴스화 (예: Kuzu, Memgraph 등)
graph_store = KuzuGraphStore(database=db)

# 2. 문서에서 속성 그래프 자동 추출 및 인덱스 생성
index = PropertyGraphIndex.from_documents(
    documents,
    property_graph_store=graph_store,
    llm=llm,
    embed_model=embed_model,
    show_progress=True
)

# 3. Vector 검색과 Graph 탐색이 결합된 하이브리드 리트리버 생성
retriever = index.as_retriever(
    include_text=True,    # 노드의 원본 텍스트 컨텍스트 포함
    similarity_top_k=5,   # 벡터 유사도로 찾을 시작 노드 수
    explore_direction="BOTH" # 엣지 탐색 방향 지정
)

# 4. 쿼리 실행
query_engine = index.as_query_engine(retriever=retriever)
response = query_engine.query("Neo4j와 Kuzu의 차이점을 설명해줘.")
